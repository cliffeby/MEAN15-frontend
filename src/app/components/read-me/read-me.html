Rochester Golf Management System - Project Overview

Instructions

Create a Match

  1. Open Matches from the sidebar.
  2. Click New Match (plus button) to open the match form.
  3. Enter or accept the default match name. (Default is configured in Admin → Default Match Name.) It is followed by the week number for the year. If you have multiple matches each week, follow it with an a, b, c,...
  4. Choose a Scorecard (course/tees). The selected scorecard drives pars and hole index for the course. Tees are determined by defaults set in the members profile.
  5. Set the match date.
  6. Add players by opening the Member Selection dialog.
  7. Create pairings.
  8. Save the match. The match appears in the list and is ready for scoring or editing.

Print Scorecards and Score a Match

  1. From the Matches list, open the match and click Print Scorecards. Scorecard should show the course, pairings, and strokes - both to par and off the low index player. The players handicap and tees should be listed next to the name.
  2. From the Matches list, open the match and click Enter Scores. Score entry will be either detailed hole by hole score or simple score. In the simple score mode, the score shuld be adjusted to reflect any adjustments. In the detailed score mode, adjustments are automatically handled based of the adjustment rules in the Configuration Tab
  3. The UI displays pars and applies handicaps; verify net scores and totals as you go.
  4. Save individual player scores as you complete them, or use Save All if available.
  5. When all scores are entered, mark the match Completed. This finalizes results, updates handicaps triggers audit logging, and updates any downstream calculations.
  6. To correct a score later: open the match → edit the player's entry → change hole values → save. Corrections are audited.

Add a Member

  1. Open Members from the sidebar.
  2. Click New Member to open the add form.
  3. Fill required fields (First Name) and recommended fields (Email, GHIN/rochIndex).
  4. Assign default scorecards and save.
  5. New members become selectable when creating or editing matches.

Scoring & Updating Records

  1. From Matches, open the match and click Enter Scores or open the match's scorecard view.
  2. Select a player (or open each player row) and enter hole-by-hole stroke counts.
  3. The UI displays pars and applies handicaps; verify net scores and totals as you go.
  4. Save individual player scores as you complete them, or use Save All if available.
  5. When all scores are entered, mark the match Completed. This finalizes results, triggers audit logging, and updates any downstream calculations (rochIndex updates if enabled).
  6. To correct a score later: open the match → edit the player's entry → change hole values → save. Corrections are audited.

Overview & Features

The Rochester Golf Club Management System is for golf groups that want to maintain their own handicap system. It is designed for groups of fourball matches played on the same day, but can be used for other formats.

Using selected members, the "admin" can create matches, foursomes and pairings. Based on member handicaps, it can create a marked up scorecard for a one- and two-ball group match and fourball(nassau).

Admin entered scores are then used to generate handicaps and reports.

Access to the system is controlled via user roles and permissions assigned to an individual via email. A "user" is a view only role, while an "admin" can create matches, add members, and manage settings. Other roles are "developer" and "field-hand."

Dashboard

The Dashboard provides a consolidated, at-a-glance view of key metrics and recent activity: totals, trends, and quick actions for matches, members, scorecards, and scores.

Members Features

The Members section allows you to create and manage player profiles. including contact information, handicap, indexes, and default course/tee preferences. The member list is adjustable to show selected columns. These column selections and pagination options (like many others in the system) are persisted by user.

To maintain data integrety, member records shouldnot be deleted if the member has played in a match Instead, you can mark a member as hidden to hie them from selection in matches and reports while preserving historical data.

The default course/tee preferences on the member profile are used to pre-select tees when for a member. Course handicaps are adjusted based on these preferences.

Scorecards Features

The Scorecards section contains the data for courses and tees. Grouping is based on Course Name,so the selected name should stay consistent across all defined tees. Minimum data for each course/tee combination is par, rating, slope, name/abreviation, and hole handicap values. The handicap values are needed to create a preconfigured scorecard for the match. Par detail values are needed to calculate adjusted scores by hole when detailed score entry is used.

Matches Features

The Matches section is where you create and manage golf matches.
- Add Match: This create a new match by selecting players, course/tee (scorecard),
and date. Pairings and foresomes are based on player handicaps using an A and B player
method with aggregate two-man team hanicapsas balanced as possible. The match name is partially
pre-filled based on a default value and the week number, but can be edited.
Use the buttons to add members and create pairs and foursomes. ReMix is not yet implemented, but you can...

The match date defaults to the current date.

Once a match is created, you can print scorecards and enter scores. Scorecards are pre-populated with player names, tees, handicaps, and pars. Score entry can be either simple (total score) or detailed (hole-by-hole).

When a match is completed, handicap record are created and the playing members' handicaps are updated.

Scores Features

The Scores section allows you to view and manage individual score entries. You can filter scores by member, match, or date range. Score details include hole-by-hole breakdowns, net scores, and adjustments. You can edit scores to correct errors or update handicaps, with all changes logged for audit purposes.

HCaps Features

The HCaps section provides a view of player handicaps and indexes. Columns are selectable and search can focus on a single player history

Reports Features

The Reports section offers a listing of current index vaules sorted by date played, a roster report showing the most recent match with players grouped by partner and foursome, and a winners report showing the top 5 net scores for the past 12 months. Other reports can be genrated on request.

Email Features

The Email section allows you to manage email templates, notifications, and communication settings. You can create, edit, and send emails to members, configure automated notifications, and customize email content.

Users Features

The Users section allows you to manage user accounts, roles, and permissions. You can create new user accounts, assign roles (e.g., admin, user), and control access to different features of the application based on these roles. Developer has full access. It also sees the code documents. Admin has access to all features but is relieved from seeing the code documents. Fieldhand can create matches and add members. It cannot delete anything. Fieldhand and User roles do not have access to the Users to for changing role permissions. User has view-only access to matches, members, and scores.

Application Configuration

You can configure default values for matches, set up email templates, and adjust other system-wide settings to tailor the application to your club's needs.

The audit log tracks all changes to matches, scores, and handicaps, including who made the change and when. This is useful for accountability and troubleshooting.

Dashboard Feature

Overview

The Dashboard provides a consolidated, at-a-glance view of key metrics and recent activity: totals, trends, and quick actions for matches, members, scorecards, and scores.

Frontend (Angular)

  • Components: src/app/components/dashboard/ — summary cards, charts, and activity feed.
  • Service: src/app/services/dashboardService.ts — aggregates backend metrics and recent activity.
  • Routes: /dashboard (default landing for authenticated users with appropriate roles).
  • UI: Material cards, charts (Chart.js or ngx-charts), sparklines, and quick-action buttons (create match, add member, import scores).

State & Data

  • The Dashboard loads data via `DashboardDataService` and uses `DashboardStatsService` for derived metrics.
  • It relies on local component state and Angular Signals (computed values) rather than NgRx for widget state and aggregates.
  • Services make parallel API calls and normalize responses for the UI; caching/refresh logic lives in services or the component.
  • Note: other parts of the app use NgRx; the Dashboard intentionally uses lightweight services/signals for quick rendering.

Backend (Node.js/Express/Mongoose)

  • Endpoints: GET /api/dashboard/summary, GET /api/dashboard/trends, GET /api/dashboard/recent — return aggregated counts, time-series, and recent activity.
  • Implementation: Lightweight aggregation queries or precomputed metrics (cron jobs) for large datasets to keep response times low.
  • Security: Role-based access control to restrict sensitive metrics to admins.

Interactive Widgets & UX

  • Summary cards for totals (members, matches, active scorecards) with drill-through links to lists.
  • Time-series charts for trends (matches per month, scores submitted) with range selectors.
  • Recent activity feed with clickable items (recent matches created, imports, admin actions).
  • Quick actions: buttons to create a match, open member form, or start an import workflow.

Notifications & Performance

  • Live-refresh or manual refresh options; loading skeletons while widgets load.
  • Cached responses and incremental updates to avoid excessive backend load.

Scorecards Feature

Overview

The Scorecards feature defines and manages templates used to capture hole-by-hole score data for matches. Scorecards map course holes, pars, yardages and provide the structure for score entry and reporting.

Frontend (Angular)

  • Components: src/app/components/scorecard-list/, scorecard-form/, scorecard-edit/
  • Service: src/app/services/scorecardService.ts
  • Model: src/app/models/scorecard.ts
  • Routes: /scorecards, /scorecards/new, /scorecards/edit/:id
  • UI: Material forms for creating/editing scorecards, preview mode for hole layout, and assignment controls to attach scorecards to matches.

NgRx Store Usage

  • Actions: Load, create, update, delete scorecards
  • Reducers: Manage scorecards collection and selected template state
  • Selectors: Query available scorecards and templates for match assignment
  • Effects: Handle API calls for scorecard CRUD and template imports/exports

  • Endpoints: GET /api/scorecards, GET /api/scorecards/:id, POST /api/scorecards, PUT /api/scorecards/:id, DELETE /api/scorecards/:id
  • Model: backend/models/Scorecard.js — stores holes array, par values, yardages, and metadata.
  • Validation: Ensures required hole definitions and consistency for front-end rendering and score entry.

Form Handling & Validation

  • Reactive forms for scorecard templates with validation for hole count and par/yardage numeric ranges.
  • Preview and import features for common course templates (CSV/JSON import).

Interactive Tables & UI

  • Browse and search scorecard templates, preview hole layouts, and assign a template to a match when creating matches.
  • Export/import of scorecard templates for sharing between environments.

Notifications & Feedback

  • Confirmation for template deletion and snackbars for create/update operations.
  • Validation errors displayed inline in forms during template creation.

Matches Feature

Overview

The Matches feature enables users to create, view, edit, and manage golf matches, including player lineups, scorecards, and match status. It is tightly integrated with NgRx for state management in the frontend.

Frontend (Angular)

  • Location: Components: src/app/components/match-list/, match-form/, match-edit/, match-selection-dialog/
  • Service: src/app/services/matchService.ts
  • Model: src/app/models/match.ts
  • Store: src/app/store/match/ (actions, reducers, selectors, effects)
  • Routes: /matches, /matches/edit/:id, /matches/new
  • List, add, edit, delete, and view matches
  • Player lineups, scorecard assignment, match results
  • Match selection dialog for other features (e.g., score entry)
  • Access via sidebar: Click "Matches" to view the list
  • Data is managed via NgRx store and fetched from the backend via MatchService

NgRx Store Usage

  • Actions: Define match-related events (load, create, update, delete)
  • Reducers: Manage match state (list, selected match, loading, error)
  • Selectors: Query match state for components (get all matches, get selected match)
  • Effects: Handle side effects (API calls for matches)
  • Benefits:
    • Centralized state management for matches
    • Automatic UI updates when match data changes
    • Simplified data flow and debugging
    • Decouples API logic from UI components

Backend (Node.js/Express/Mongoose)

  • Location: Model: backend/models/Match.js, Controller: backend/controllers/matchController.js, Routes: backend/routes/matchRoutes.js
  • API Endpoints:
    • GET /api/matches — List all matches
    • GET /api/matches/:id — Get match by ID
    • POST /api/matches — Add new match
    • PUT /api/matches/:id — Update match
    • DELETE /api/matches/:id — Delete match
  • Data Model: Fields: name, date, scorecardId, lineUps, status, results, etc.
  • Validation and business logic handled in controller/model

Golf Management (CRUD)

  • Manage matches, scorecards, and member profiles with full CRUD operations.
  • Create matches with player lineups, assign scorecards, and schedule dates.
  • Assign and manage scorecards used for score entry and reporting.
  • Edit match, scorecard, and member details with validation and audit logging.
  • View match and member histories in searchable, filterable, and paginated tables.

HCap Feature

Overview

The HCap (rochIndex) feature manages member handicaps, adjustments, and historical rochIndex records. It supports importing/updating handicaps, applying rochIndex rules for matches, and tracking rochIndex changes over time.

Frontend (Angular)

  • Components: src/app/components/hcap-list/, hcap-form/, hcap-history/
  • Service: src/app/services/hcapService.ts
  • Model: src/app/models/hcap.ts
  • Routes: /hcaps, /hcaps/upload, /hcaps/history/:memberId
  • Actions: Load current hcaps, import/update handicaps, load history (invoked via HCapService, not NgRx)
  • State: Managed by HCapService and component-local state (no NgRx feature)
  • UI: Tables for current handicaps, per-member history view, upload/import UI for rochIndex files, and controls to recalculate handicaps.

NgRx Store Usage

  • Actions: Load current hcaps, import/update handicaps, load history
  • Reducers: Manage hcap state, loading indicators, and error states
  • Selectors: Query current rochIndex values, historical traces, and pending import results
  • Effects: Handle API calls for importing, recalculation, and historical data retrieval

Backend (Node.js/Express/Mongoose)

  • Endpoints: GET /api/hcaps, POST /api/hcaps/import, GET /api/hcaps/:memberId/history, PUT /api/hcaps/:memberId
  • Model: backend/models/HCap.js — stores current rochIndex, history entries, and metadata about calculation rules.
  • Validation: Ensures imported files match expected formats and values fall within acceptable ranges.

Form Handling & Validation

  • Upload form for rochIndex files with client-side format validation and preview of parsed records.
  • Server-side checks to validate member references and numeric ranges before committing imports.

Interactive Tables & UI

  • Filterable list of members with current rochIndex, last update date, and source (manual/import).
  • Per-member history view showing previous rochIndex values, reasons, and timestamps.
  • Recalculation tools for administrators to apply new rules or backfill historical data.

Notifications & Feedback

  • Progress indicators for large imports and snackbars for import/validation results.
  • Detailed error reports for failed import rows with recommended remediation steps.

Project Overview

Tech Stack & Backend Setup

  • Frontend: Angular (see frontend/package.json for exact version)
  • Backend: Node.js with Express.js
  • Database: MongoDB using Mongoose ODM (optional Azure Cosmos DB deployments supported)
  • Authentication: Microsoft Entra (MSAL) with JWT validation (JWKS)
  • Password Security: bcrypt for password hashing where local credentials are used
  • Environment (example): Angular CLI: 20.3.14 · Node: 22.20.0 · npm: 11.9.0 · OS: win32 x64
  • API Endpoints: RESTful endpoints for members, matches, scores, scorecards, users, hcaps, orphans, audit, email, auth, and config
  • Middleware: Helmet for security headers, CORS, rate limiting, compression, and centralized error handling
  • Logging: Morgan for request logging (development) and audit logs for admin actions
  • Deployment Ready: Modular route/controller structure, environment-based config, and production-ready error handling
